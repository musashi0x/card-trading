# ✦ TopDeck — non-custodial trading card marketplace

A trading card marketplace where **the payment flow is the product**. Cards are
on-chain Stellar assets; trades settle through a single Soroban escrow/settlement
contract. Built for the **Consumer & Merchant Payment Flows** track.

> Chain is the source of truth for ownership and money. Postgres is a
> read-optimized mirror for search, metadata, and history.

## Why it's interesting (payment flow)

- **Non-custodial escrow** — a buyer's offer locks USDC inside a Soroban
  contract, not a company wallet.
- **Atomic settlement** — on acceptance, the card goes to the buyer and USDC
  (minus a transparent platform fee) goes to the seller in **one transaction**.
  All-or-nothing.
- **Consumer protection** — an open offer can be withdrawn at any time before the
  merchant accepts; funds were never at risk.
- **One primitive, two flows** — buy-now is just "an offer at the asking price
  that auto-accepts," so both UX paths share the same settlement code.

```
 Merchant lists card ──▶ card locked in contract
 Consumer makes offer ──▶ USDC locked in contract
 Merchant accepts ──▶ ATOMIC: card→buyer · USDC−fee→seller · fee→platform
 (or) Consumer withdraws ──▶ USDC returned, no risk
```

## Quick start

Prereqs: Node 20+, pnpm, Docker, Rust + `wasm32` target, `stellar` CLI, Freighter.

```bash
pnpm install

# 1. Postgres (host port 5433) + Kubo IPFS node (RPC :5001, gateway :8080)
docker compose up -d

# 2. Bootstrap testnet: fund accounts, issue test USDC + cards
pnpm stellar:setup            # writes stellar-accounts.json

# 3. Build + deploy the contract, deploy SACs, init
pnpm contract:build
pnpm contract:deploy          # writes deploy.json

# 4. Configure env (CONTRACT_ID, issuer, etc.)
#    Copy values printed by the steps above into .env (see .env.example)
cp .env.example .env          # then fill in, or generate from the json artifacts

# 5. Database
pnpm db:migrate
pnpm db:seed                  # merges card metadata with on-chain issuer + SAC

# 6. Demo data + run
pnpm --filter @cardmkt/scripts run demo   # a few open listings (needs API up)
pnpm dev                                   # web :3000, api :4000
```

Import the **merchant** and **consumer** secret keys from `stellar-accounts.json`
into Freighter (set network to *testnet*) to play both sides.

## Verify it works (headless, against testnet)

```bash
pnpm --filter @cardmkt/api run dev          # in one terminal
pnpm --filter @cardmkt/scripts run e2e      # in another
```

Drives the full build → sign → submit flow with the demo keys and asserts:
- offer → accept settles atomically with the 2% fee
- buy-now transfers the card at asking price
- withdraw refunds the buyer's escrowed USDC

Contract unit tests: `cd packages/contracts && cargo test` (6 tests).

## The 2-minute demo

1. **Merchant** connects wallet, lists *Nova Dragon* for 50 USDC → card locked
   on-chain (show it in the explorer — the contract holds it).
2. **Consumer** connects, browses, offers 40 USDC → USDC locked in the contract.
3. Show the explorer: the funds are in the **contract**, not the seller.
4. Merchant clicks **Accept** → one atomic tx: card→buyer, 39.2 USDC→seller,
   0.8 USDC→platform.
5. Open **Trades**: click the settlement hash → verify on the block explorer.
6. **Protection beat:** Consumer makes another offer, then **Withdraws** it →
   funds returned, never at risk.

## Contract interface

```
init(admin, platform, usdc_token, fee_bps)
list(seller, card_token, price) -> listing_id     # locks the card
cancel_listing(seller, listing_id)                # returns the card
make_offer(buyer, listing_id, amount) -> offer_id # locks USDC
withdraw_offer(buyer, offer_id)                   # refunds USDC
accept_offer(seller, offer_id)                    # atomic settle
buy_now(buyer, listing_id)                        # atomic settle @ asking price
```

Deployed addresses live in `deploy.json`; demo accounts in `stellar-accounts.json`
(both git-ignored, regenerated by the setup/deploy scripts).

## Card art on IPFS

Uploaded card photos are pinned to IPFS at mint time and stored in Postgres as
`ipfs://<CID>` (CIDv1); the API resolves them to gateway URLs in every response,
so the web app just renders normal `https` image URLs. Already-hosted `https:`
images and legacy inline `data:` values keep working untouched.

- **Local dev**: `docker compose up -d` starts a Kubo node; point
  `IPFS_API_URL=http://localhost:5001` and `IPFS_GATEWAY_URL=http://localhost:8080`
  at it (see `.env.example`).
- **Production**: leave `IPFS_API_URL` empty and set `PINATA_JWT` (a
  [Pinata](https://pinata.cloud) API JWT) plus `IPFS_GATEWAY_URL` to your
  dedicated gateway.
- **Neither configured**: minting falls back to storing the uploaded data URL
  inline (a warning is logged at API startup).

## Notes / scope

Test USDC and cards are testnet assets (clearly labeled). Out of scope for the
hackathon: physical shipping, fiat anchors (SEP-24), multi-asset path payments,
production indexer streaming.

## Troubleshooting

- **Minting fails when connected as the platform issuer.** An issuer cannot hold
  a trustline to an asset it issues, so minting/distributing copies *to the
  issuer account itself* is impossible (the classic flow would build a
  self-trustline `changeTrust` that Stellar rejects). Connect a **different**
  wallet — any funded `G…` account that isn't `PLATFORM_ISSUER`, or a passkey
  `C…` smart wallet — as the card owner. The API now rejects this up front with
  `OWNER_IS_ISSUER`.
- **Listing right after minting briefly errors, then succeeds.** The API
  pre-flights the seller's balance against **Horizon** but simulates the contract
  call against the **Soroban RPC**, which ingests new state a ledger or two
  later. Right after a mint+distribute the SAC `transfer` simulation can revert
  with "trustline entry is missing"; `buildContractTx` retries this lag and a
  persistent miss surfaces as a `MISSING_TRUSTLINE` 400 rather than a 500.
